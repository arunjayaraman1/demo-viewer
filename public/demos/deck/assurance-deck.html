The problem

Enterprise QA is a governed engagement —
run on ungoverned, fragmented tooling.

Teams buy testing as a service — scoped, delivered, signed off, billed. But the work underneath is stitched together by hand, tool by tool, with no consistent line of oversight.

Fragmented

Every tool, wired by hand

Jira, GitHub, BrowserStack, the test framework — each integrated separately, per team, per project. Cloned boilerplate drifts apart.

Manual

Authoring & upkeep by hand

Cases hand-written, automation hand-coded, breakages hand-fixed. The cost scales with every feature and every browser.

Ungoverned

Passing ≠ assured

No consistent gates, evidence trail, or sign-off. Change the tool and the oversight changes with it — lock-in decides governance.

And there are always two sides

The demo — and most tools — only show one. A governed service has a consumer who asks, and a provider who delivers.

Consumer

The app team

Raises a work order: "assure my feature, tell me it's safe to release." Wants scope, a health view, and a sign-off — not a script generator.

Provider

The assurance team

Scopes it, prices it, delivers & evidences it. Runs the engine. Needs it to work the same way across every team's stack.

Layered architecture05

The Assurance Agent, up close

Agent skills & MCP, abstracted — and integrated into custom enterprise workflows.

The portable plugin

One unit that drops into any host

/assure-context
Consolidate Jira epic + docs into one context

ISTQB Gherkin — happy, validation, edge
/assure-automate
Generate runnable tests for the adapter
/assure-heal
Triage drift vs regression; propose a fix
/assure-pipeline
Wire the CI matrix — browser × OS × auth
/assure-flaky
Detect, classify, quarantine flaky tests
Tool-agnostic core
One authored intent → consolidated context, ISTQB Gherkin, and a capability profile that selects the adapter. Proven by generating both Playwright & pytest-bdd from one Gherkin set.

MCP is the lingua franca

Every host speaks MCP, so the same capabilities drive the flow anywhere. The /assure-* agent skills are the Claude-native ergonomic layer bundled alongside.

Host- & framework-independent

The host provides the model + tool loop; the plugin provides the assurance behaviour. The capability profile picks the framework — no logic changes.

Little to build, much reused

Agent, agent skills, adapters & MCP wiring exist and run green. New is only a versioned plugin package + a thin push/read API. The platform carries zero vendor clients.

HITL governance · gates A–F07

Same engine, two modes — the business case

Meets teams where they are. Governs them the same way.

Assisted · human at the wheel

A QA in Claude Code / Cursor

Runs /assure-* interactively. HITL is the host's tool-approval prompts plus the platform gates. This is the demo's mode.

Autonomous · in CI

The Claude Code GitHub Action

Runs the same agent headlessly on a trigger — PR opened, nightly, story moved to Done. Risk-bearing transitions still gate and wait for a human.

Cycle time cut
What took most of a sprint — authoring, automation, triage — now runs in hours. Features reach Assured the same day they're dev-done.
QA effort reduced
The engine drafts the cases, writes the automation and triages the failures. QA reviews and signs — instead of hand-building each one.
Scales without headcount
One QA governs many features across teams. Coverage grows with the portfolio, not with the org chart.
Portfolio-wide governance
Every feature, team and run under one control plane — the same gates, evidence and audit, end to end.
